# GTCRN：轻量级实时语音降噪

## 概述

GTCRN（Grouped Temporal Convolutional Recurrent Network）是一个面向实时语音降噪的轻量级网络。设计目标是在极低参数量（~140K）下达到可用的降噪效果，同时支持流式推理（延迟 10ms）。

核心思路是**复数掩码（CRM）**：在 ERB 频带上学习一个复数比率掩码，同时修正幅度和相位，乘回原始频谱得到干净语音。

整体是一个 U-Net 结构：编码器压缩频率维度提取抽象特征，DPGRNN 建模时频依赖，解码器恢复频率维度生成掩码。

---

## 一、为什么用 ERB 频带

GTCRN 的第一步是把 513 个线性频点压缩到 219 个 ERB 频带。

**线性频谱的问题**：STFT 在每个频率上分配相同的分辨率，但人耳不是这样工作的。人耳对低频（100-1000Hz）极其敏感，对高频（4000Hz 以上）相对迟钝。用线性频谱做语音增强，大量计算浪费在人耳感知不到的高频细节上。

**ERB 的解决方案**：按人耳感知特性重新分配频率分辨率。低频保留精细的频点，高频合并成宽频带。513 个线性频点压缩到 219 个 ERB 频带，计算量直接减少一半以上，同时不损失感知质量。

**为什么不用 Mel**：Mel 和 ERB 都是感知频率尺度，在语音增强任务上 ERB 效果略好，且公式更简洁。

---

## 二、整体架构：为什么是 U-Net

语音降噪是一个**输入输出同尺寸**的任务：输入带噪频谱，输出同尺寸的干净频谱（或掩码）。这个特性天然适合 U-Net。

```
输入频谱 [B, 513, T, 2]
    │
    ▼ ERB 压缩
[B, 3, T, 219]  ← 幅度 + 实部 + 虚部
    │
    ▼ SFE：频带局部聚合
[B, C, T, 219]
    │
    ├──────────────────────────────────────────────────────┐
    │ Encoder                                              │
    │   219 ──────────────────────────── skip1 ──────────→│
    │     ↓ stride=2                                       │
    │   110 ──────────────────────────── skip2 ──────────→│
    │     ↓ stride=2                                       │
    │    55 ──── GTConvBlock×6 ──── skip3~8 ─────────────→│
    │     ↓                                                │
    │   DPGRNN×2：时频双路径建模                            │
    │     ↓                                                │
    │ Decoder                                              │
    │    55 ← GTConvBlock×6 + skip3~8                      │
    │     ↑ stride=2                                       │
    │   110 ← + skip2                                      │
    │     ↑ stride=2                                       │
    │   219 ← + skip1                                      │
    └──────────────────────────────────────────────────────┘
    │
    ▼ ERB 还原
[B, 2, T, 513]  ← 复数掩码
    │
    ▼ CRM × 输入频谱
输出干净频谱 [B, 513, T, 2]
```

**U-Net 的核心价值**：编码器在压缩过程中会丢失细节（边界、精细频谱结构），skip connections 把这些细节直接传给解码器，让解码器不需要从抽象特征中重新"发明"这些细节。

**为什么需要压缩**：不压缩就没有"抽象"。网络需要先把 219 个频带的局部信息压缩成全局的语义表示（"这段是语音"、"这段是噪声"），才能做出正确的判断。直接在 219 维上操作，网络只能看到局部，无法建立全局理解。

---

## 三、输入处理：为什么拼接幅度、实部、虚部

GTCRN 的输入不是直接用复数频谱，而是拼接三个通道：幅度谱、实部、虚部。

**幅度谱的作用**：幅度谱直接反映能量分布，是判断"这里有没有语音"最直观的特征。网络如果只看实部和虚部，需要自己计算 $\sqrt{re^2 + im^2}$，增加了学习难度。显式提供幅度谱相当于给网络一个"捷径"。

**实部和虚部的作用**：保留相位信息。复数掩码需要同时修正幅度和相位，所以网络必须知道原始的相位状态。

**为什么不用幅度谱 + 相位谱**：相位是角度，是周期性的（$-\pi$ 和 $\pi$ 是同一个值），神经网络处理周期性数值很困难。实部+虚部是笛卡尔坐标，没有这个问题。

---

## 四、SFE：为什么需要频带局部聚合

SFE（Subband Feature Extraction）在进入编码器之前，把每个频带和它的左右邻居拼在一起。

**问题**：ERB 压缩后，相邻频带之间的关系被割裂了。每个频带只有自己的信息，但语音的共振峰、谐波结构都是跨频带的。

**SFE 的解决方案**：把每个频带和左右邻居拼成一组，让网络在第一层就能看到局部的频带上下文。这相当于在频率轴上做了一个小感受野的预处理，让后续的卷积层从一开始就有跨频带的视野。

**为什么不直接用大卷积核**：SFE 是固定操作（不学习），计算代价极低。用大卷积核虽然也能扩大感受野，但参数量和计算量都会增加。

## 五、编码器设计

### 5.1 两层下采样：为什么 stride=2

编码器用两层 stride=2 的卷积把频率维度从 219 压缩到 55（219→110→55）。

**为什么要压缩**：
- 减少后续 GTConvBlock 的计算量（频率维度小了 4 倍）
- 强迫网络提取抽象特征，而不是记住局部细节
- 为 DPGRNN 提供紧凑的特征表示

**为什么是 stride=2 而不是 pooling**：stride 卷积在下采样的同时还能学习特征变换，pooling 只是简单地取最大值或平均值，表达能力更弱。

**为什么压缩 4 倍而不是更多**：压缩太多会丢失太多频率细节，解码器即使有 skip connections 也难以恢复。4 倍是在计算效率和信息保留之间的平衡点。

### 5.2 GTConvBlock：设计哲学

GTConvBlock 是编码器的核心模块，连续堆叠 6 个，负责提取深层特征。

#### 为什么用 ShuffleNet 风格（只处理一半通道）

每个 GTConvBlock 把输入通道分成两半，只对一半做卷积处理，另一半直通，最后 shuffle 混合。

**设计动机**：在极低参数量的约束下，不能对所有通道都做完整的卷积。ShuffleNet 的思路是：不是每个通道都需要在每一层被处理，直通的那一半保留了上一层的信息，处理的那一半提取新特征，shuffle 后两者混合。这样用一半的计算量达到接近全量处理的效果。

**为什么 shuffle 而不是拼接**：shuffle 把两半通道交错排列，让下一层的每个卷积核都能同时看到"处理过的"和"直通的"通道，促进信息流动。简单拼接会让两半通道在空间上分离，下一层需要更大的卷积核才能融合。

#### 为什么用深度可分离卷积

标准卷积同时做空间特征提取和通道混合，参数量是 $k^2 \times C_{in} \times C_{out}$。深度可分离卷积把这两件事分开：

- **深度卷积**：每个通道独立做空间卷积，只提取空间特征
- **逐点卷积**：1×1 卷积做通道混合

参数量降低约 7 倍，但表达能力损失有限，因为空间特征和通道特征本来就是相对独立的两件事。

#### 为什么用膨胀卷积金字塔 [1, 2, 4, 8, 4, 2]

6 个 GTConvBlock 的膨胀率不是固定的，而是先扩大后收缩：

```
Block 1: dilation=1  感受野: 3
Block 2: dilation=2  感受野: 7
Block 3: dilation=4  感受野: 15
Block 4: dilation=8  感受野: 31
Block 5: dilation=4  感受野: 15
Block 6: dilation=2  感受野: 7
```

**为什么先扩大**：膨胀卷积让感受野指数增长，用少量参数覆盖大范围的频率上下文。语音的谐波结构跨越很宽的频率范围，需要大感受野才能捕捉。

**为什么后收缩**：大膨胀率会产生"网格效应"——只采样固定间隔的位置，跳过中间的频点。收缩阶段用小膨胀率填补这些被跳过的位置，让特征更连续。

**为什么对称**：对称的金字塔结构让网络在扩张阶段建立全局视野，在收缩阶段细化局部细节，两个阶段互补。

#### 为什么需要 TRA（时域门控）

GTConvBlock 内部有一个 TRA（Temporal Recurrent Attention）模块，对时间轴做门控。

**问题**：卷积在时间轴上是静态的，对每一帧的处理方式相同。但语音是动态的——语音帧和静音帧、稳态噪声帧和瞬态噪声帧，应该用不同的处理强度。

**TRA 的解决方案**：计算每帧的能量，用时序模型（GRU 或卷积）预测一个门控系数，对特征做逐帧加权。能量高的帧（语音）保留更多特征，能量低的帧（静音/噪声）抑制特征。

**为什么 V1 把 GRU 换成卷积（TRALite）**：GRU 有隐藏状态，在流式推理时需要维护状态，部署复杂。卷积版本无状态，可以直接量化部署，且效果差距不大。

#### 为什么需要 SE Block（通道门控）

SE（Squeeze-and-Excitation）Block 对通道维度做门控：先全局平均池化得到每个通道的全局统计，再用全连接层预测每个通道的重要性系数。

**作用**：不同通道学到了不同的特征（有的通道对低频敏感，有的对高频敏感），SE Block 让网络动态地调整各通道的权重，突出当前帧最相关的特征。

**TRA vs SE 的分工**：
- TRA：时间维度的门控（哪些帧重要）
- SE：通道维度的门控（哪些特征重要）

两者互补，共同提升特征的选择性。

## 六、DPGRNN：为什么需要双路径 RNN

编码器输出的特征是 [B, 32, T, 55]——32 个通道，T 帧，55 个频带。这是一个二维的时频结构，需要同时建模时间依赖和频率依赖。

### 6.1 为什么不用单路径 RNN

如果只用时间方向的 RNN，网络能学到"这段时间是语音"，但无法学到"这些频带之间有谐波关系"。如果只用频率方向的 RNN，网络能学到频带间的关系，但无法利用时序上下文。

语音降噪需要两种信息：
- **时序上下文**：语音有持续性，噪声有统计规律，需要看多帧才能判断
- **频率上下文**：语音的谐波结构、共振峰分布，需要跨频带建模

### 6.2 双路径设计

```
输入 [B, 32, T, 55]
    │
    ▼ Intra（频率方向，双向）
    │ 把每帧的 55 个频带当作序列，双向 GRU 建模频带间关系
    │ 每帧独立处理，不跨帧
    │
    ▼ Inter（时间方向，单向）
    │ 把每个频带的 T 帧当作序列，单向 GRU 建模时序依赖
    │ 每个频带独立处理，不跨频带
    │
输出 [B, 32, T, 55]
```

**为什么 Intra 双向**：频率轴没有因果约束，低频和高频互相都有帮助。双向 GRU 让每个频带都能看到所有其他频带的信息，建模更完整的谐波关系。

**为什么 Inter 单向**：时间轴有因果约束，实时系统不能看未来的帧。单向 GRU 只看过去，保证了流式推理的可行性。

**为什么堆叠两个 DPGRNN**：一个 DPGRNN 只做了一轮时频交互。第一个 DPGRNN 建立初步的时频理解，第二个在此基础上进一步细化，类似于深层网络的逐层抽象。

### 6.3 为什么用残差连接

每个 DPGRNN 的输出加回输入（残差连接）。

**原因**：DPGRNN 的任务是"调整"特征，而不是"重新生成"特征。残差连接让网络只需要学习"需要改变什么"，而不是从零开始生成新特征。这大大降低了学习难度，也让梯度更容易反向传播。

## 七、解码器设计

解码器是编码器的镜像：6 个 GTConvBlock（膨胀率反向 [2,4,8,4,2,1]）+ 两层转置卷积上采样（55→110→219）。

### 7.1 为什么是编码器的镜像

ERB 掩码生成是一个**空间重建任务**：编码器把 219 个频带压缩到 55，解码器需要恢复到 219 个频带，每个频带生成一个掩码值。

这个"压缩→恢复"的过程天然对称。解码器需要逐步恢复频率分辨率（55→110→219），每一步都需要对应分辨率的 skip connection 提供细节信息。

### 7.2 Skip Connections 的作用

```
编码器                          解码器
219 ──── skip1 ─────────────→ 219
  ↓                               ↑
110 ──── skip2 ─────────────→ 110
  ↓                               ↑
 55 ──── skip3~8 ───────────→  55
```

编码器在压缩过程中丢失了精细的频率细节（具体的频谱边界、局部能量分布）。解码器在上采样时，如果只靠抽象特征，无法知道"这个频带的掩码应该是 0.8 还是 0.3"。Skip connections 把编码器各层的特征直接传给解码器对应层，补充这些被压缩掉的细节。

**为什么需要多层 skip 而不是只用最后一层**：
- skip1 (219维)：保留了最精细的频率边界信息
- skip2 (110维)：保留了中等粒度的频带结构
- skip3~8 (55维)：保留了 GTConvBlock 各层的抽象特征

每一层 skip 提供不同抽象程度的信息，解码器在恢复时可以利用多个尺度的参考。

### 7.3 解码器的 GTConvBlock 膨胀率为什么反向

编码器：[1, 2, 4, 8, 4, 2]（先扩大后收缩）
解码器：[2, 4, 8, 4, 2, 1]（先扩大后收缩，但起点不同）

解码器的任务是从抽象特征生成精细掩码，同样需要先建立大感受野的全局视野，再用小感受野细化局部细节。膨胀率的具体排列是经验性的，核心原则不变：避免网格效应，同时覆盖多个尺度。

## 八、复数掩码（CRM）：为什么不用实数掩码

传统语音增强常用实数掩码（IBM 或 IRM），只修正幅度，不处理相位。GTCRN 用复数比率掩码（CRM），同时修正幅度和相位。

**实数掩码的局限**：
$$\hat{S}(k,t) = M(k,t) \cdot |X(k,t)|$$

只改变幅度，相位直接用带噪语音的相位。但带噪语音的相位已经被噪声污染，直接使用会引入相位失真，影响语音质量。

**复数掩码的优势**：
$$\hat{S}_r = M_r \cdot X_r - M_i \cdot X_i$$
$$\hat{S}_i = M_r \cdot X_i + M_i \cdot X_r$$

通过复数乘法，同时修正实部和虚部，等价于同时修正幅度和相位。网络可以学到"这个频带的相位应该往哪个方向修正"，而不是盲目使用带噪相位。

**为什么不直接预测干净频谱**：直接预测频谱是一个绝对值预测问题，难度更高。预测掩码是一个相对值问题（"保留多少、修正多少"），网络更容易学习。

## 九、版本演进

### V1：基线（48kHz 适配）

原版 GTCRN 是 16kHz 的，V1 适配 48kHz，同时做了算子级轻量化：

| 改动 | 原因 |
|------|------|
| ERB 用固定权重（不学习） | ERB 变换是固定的感知尺度，不需要学习，省参数 |
| 卷积改为深度可分离 | 参数量减少约 7 倍，效果损失有限 |
| TRA 的 GRU 改为卷积 | 无状态，便于部署和量化 |
| DPGRNN 加低秩投影 | RNN 参数量正比于隐藏维度的平方，低秩投影显著减少参数 |

**核心原则**：数据流不变，只换算子。这样可以精确评估每个改动的收益，不引入架构变量。

### V2：瞬态感知损失

**问题**：标准损失函数对每帧权重相同。键盘敲击、鼠标点击这类瞬态噪声只占少数帧，损失贡献小，网络学不好。

**方案**：不改网络，只改损失函数。检测能量突变帧（瞬态帧），给这些帧更高的损失权重（5倍）。

**为什么不改网络**：能用损失函数解决的问题，就不动网络。改网络会增加推理开销，改损失函数推理零开销。

**代价**：训练时间变长（29→71 epochs），因为瞬态帧的高权重让优化更困难。

### V3：因果化（实时推理）

**问题**：V1/V2 处理第 t 帧需要看整段音频，无法实时使用。

**根本原因**：卷积的对称 padding 让每帧都能看到未来的帧。

**方案**：把所有时间轴上的对称 padding 改为单侧 padding（只在左边 pad，右边不 pad）。这样每帧只能看到过去的帧，满足因果约束。

```
非因果卷积（k=3, d=2）：
  t-4  t-3  t-2  t-1   t   t+1  t+2
            ●         ○         ●     ← 看了未来的 t+2

因果卷积（k=3, d=2）：
  t-4  t-3  t-2  t-1   t   t+1  t+2
   ●         ●         ○         ← 只看过去
```

**为什么 V3 把卷积核从 3×3 改为 5×5**：因果化后，每个卷积核只能看左边，感受野减半。用更大的卷积核补偿损失的感受野。

**代价**：DNSMOS 从 3.15 降到 2.98。因果模型看不到未来，信息量本来就少，这个损失是预期内的。

**DPGRNN 不需要改**：Inter（时间方向）本来就是单向的，天然因果。Intra（频率方向）在频率轴上没有因果约束，双向不需要改。

## 十、设计总结

GTCRN 的设计哲学可以归纳为几个核心决策：

**1. ERB 对齐感知**：在感知频率尺度上操作，而不是线性频率尺度。计算量减半，感知质量不损失。

**2. U-Net 处理空间重建**：掩码生成需要恢复频率分辨率，U-Net 的多层 skip connections 提供不同尺度的细节信息，是这类任务的自然选择。

**3. 双路径 RNN 处理时频依赖**：时间轴和频率轴的依赖性质不同（因果 vs 非因果，时序 vs 谐波），分开建模比混合建模更高效。

**4. 多维度门控**：TRA（时间）+ SE（通道）+ SubbandAttention（频率）三个维度的注意力机制，让网络能动态地聚焦于最相关的特征。

**5. 能用损失函数解决就不动网络**：V2 的瞬态感知改进只改了损失函数，推理零开销。这是工程上的重要原则。

**6. 因果预留**：Inter RNN 从一开始就是单向的，V3 因果化只需要改 padding，不需要重新设计架构。这说明好的架构设计应该预留扩展空间。
